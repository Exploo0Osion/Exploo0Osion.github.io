VT And EPT

VMX 根模式可以访问VMX指令，并能运行基于硬件的虚拟机。

VMX 非根模式无法访问VMX指令

Virtual Machine Monitor (VMM)是一种创建并运行虚拟机的计算机软件、固件或硬件。运行一个或多个虚拟机的虚拟机称为主机（host），每个虚拟机称为访客机（guest）。

  • 多进程内核创建多个进程 ，并安排它们的内存和执行，使它们之间无法相互干扰。
  • 虚拟机监视器（VMM）创建多个虚拟机来运行软件，并安排它们的内存和执行，使它们之间无法相互干扰

VM-Entry ：VMM 执行 VMLAUNCH/VMRESUME 指令，CPU 从根模式切换到非根模式，加载 Guest 状态，开始运行 Guest OS；

VM-Exit ：Guest 中触发了预设的敏感事件（比如 EPT 权限违例、CPUID 指令、MSR 访问等，诸如 CPUID, MOV from/to CR3），CPU 自动：

  • 把 Guest 的完整上下文保存到 VMCS；
  • 加载 Host 状态，切换到根模式，跳转到 VMM 的 VM-Exit 处理函数；
  • 把退出原因、违例信息填充到 VMCS 对应字段；
    VMM 处理完事件后，再次执行 VM-Entry，回到 Guest 继续运行。

软件通过执行VMXON指令进入VMX操作。

VMCS，在guest下，每个 vCPU 对应一个 VMCS（Virtual-Machine Control Structure，虚拟机控制结构），这里面保存了很多信息，可以用于触发事件进入到VMM。

虚拟化下的访存

HPA (Host Physical Address), HVA (Host Virtual Address)

GPA (Guest Physical Address), GVA (Guest Virtual Address)

SPT (Shadow Page table), EPT (Extended Page Table)

虚拟机中的进程访存，要实现GVA -> GPA -> HVA -> HPA的转化。即要经历以下步骤：

  1. 客户端虚拟地址经过guest中的页表转化为客户端的物理地址；
  2. 客户端的物理地址经过转换，变为宿主端的虛拟地址。VMM通常会用内部数据结构来记录这个映射。
  3. 宿主端的虛拟地址再经过host上的页表的转换变为真实可用的物理地址。

在Guest中所有页表相关的内存区域会被VMM设置为不可读不可写，那么 Guest每次访问页表时都会触发vmexit，VMM把转换GPA成HVA，再通过访问VMM对应的页表得到HPA，最后通过HPA指向的内存数据，得到下一级页表的guest物理地址。

在系统vmexit, vmentry的过程中，硬件会根据内存中VMCS结构中保存的host/guest CR3值更新CR3寄存器的值。

上述过程费时还费性能，目前有如下解决方式：

影子页表（SPT）

实现GVA->HPA映射

当guest操作系统在把当前进程的页表基址载入CR3时，VMM会截获这条指令，并将影子页表基地址写入VMCS对应的GUEST_CR3字段。再次进入guest时，硬件会把影子页表的基地址载入CR3。此时，guest页表维护的是影子页表，CR3的值指向影子页表的基地址。

Guest在访问虚拟地址时（GVA），由于CR3指向影子页表，影子页表中维护了GVA->HPA映射，MMU就可以直接读取GVA对应的HPA值，不需要VMM参与。

拓展页表（EPT）

影子页表的内存开销也很大，需要为每个Guest进程对应的页表都维护一个影子页表。

在硬件上支持GVA->GPA->HPA的两次地址转换，都由CPU硬件自动完成。

Guest CR3 指向的L4页表(GPA)，CPU需通过EPT页表来实现Guest CR3 GPA->HPA的转换。首先会查看硬件的EPT TLB，如果没有对应的转换，则会进一步查找EPT页表；如果还没有，CPU则拋出EPT Violation异常由VMM来处理 。

获得L4 页表地址后(HPA)，CPU根据GVA对应的L4页表项的内容，获取L3 页表项的GPA。如果L4页表项显示为“缺页”（Page Fault）， 直接交由Guest kernel处理（这里不会有VM-EXIT）。利用EPT访存不需要切换CR3，也不需要GPA->HVA的转化过程。
