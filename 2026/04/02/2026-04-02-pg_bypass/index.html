从kurasagi项目源码看PG bypass

PatchGuard

检查内容：IDT、GDT、MSR、SSDT、内核栈、全局变量、部分关键代码，以及 PG 自己的执行面。

初始化

KeInitAmd64SpecificState

当 InitSafeBootMode == 0 时，直接计算一个由 KdPitchDebugger | KdDebuggerNotPresent 派生的除法表达式。

当 InitSafeBootMode != 0 时，还要额外看 KiSwInterruptPresent()：

  • 如果 KiSwInterruptPresent() >= 0，仍然走那段除法表达式。
  • 如果 < 0，则直接返回错误状态，不再继续。

当除法错误时，会触发异常，异常最终仍然会落到 KiDivideErrorFault -> KiExceptionDispatch，再把控制流送往 KiFilterFiberContext。

ExpLicenseWatchInitWorker

根据下方字段的值“非 SafeBoot、无调试器、4% 概率”来判断是否进入 KiFilterFiberContext

  • KUSER_SHARED_DATA.KdDebuggerEnabled >> 1
  • InitSafeBootMode
  • KiSwInterruptPresent()
  • (__rdtsc() >> 3) % 100 <= 3 其概率约为4%

PatchGuard 上下文结构

PatchGuard 将 CmpAppendDllSection 的代码复制到其结构中，并用它通过与随机密钥的异或解密其余部分

在 PatchGuard DPC 执行过程中，引用了如 KiWaitAlways 和 KiWaitNever 等全局变量，用于编码或解码指针。

随后紧接着填入大量函数指针，许多 ntoskrnl.exe 的指针也被复制到 PatchGuard 上下文中，使 PatchGuard 能够调用函数而无需依赖内核导出表，后面还建立一系列待校验描述符数组。

kurasagi实现

kurasagi源码入口很直接：

  • DriverEntry 初始化运行时变量后，调用 BypassPatchGuard()。
  • BypassPatchGuard() 依次执行：
    • Timer::DisableAllTimers()
    • Context7::DestroyContext7()
    • Misc::DisableMiscRoutines()
    • Barricade::SetupBarricade()

也就是说，项目作者自己就把 PG 拆成了这四个对抗面。

Timer kill

主逻辑位于 kurasagi/Module/Timer.cpp。

IsPatchGuardTimer() 通过两个特征识别目标定时器：

  1. DeferredContext 不是 canonical address。
  2. DeferredRoutine 指向 CcBcbProfiler / CcBcbProfiler2。它使用例程 CcBcbProfiler 设置 DPC

DisablePatchGuardTimers() 调用 KeGenericCallDpc，该例程会导致指定例程同时在所有处理器上运行执行 DisablePatchGuardTimersIdPc()

该函数清理每个 CPU 的定时器表 KPRCB.TimerTable.TimerEntries[512]，识别到PatchGuard计时器后把 currentTimer->Dpc 改写为“加密后的 NULL”，此外它还会额外清空：

  • KPRCB->HalReserved[7] 用于存储PatchGuard timer DPC
  • KPRCB->AcpiReserved

Kill Software Interrupt

主逻辑位于 kurasagi/Module/Context7.cpp。

DestroyContext7() 有两个动作：

  1. 将 KiSwInterruptDispatch 开头改成直接ret
  2. 将 MaxDataSize 指向的位置清零

KiSwInterruptDispatch由函数 KiSwInterrupt调用，可能是由软件中断触发的机制，因为 MaxDataSize 相关全局变量会被这个路径引用，MaxDataSize也在PatchGuard的上下文结构中。

破坏辅助例程

DisableMiscRoutines() 当前只真正做了一件事：把 KiMcaDeferredRecoveryService 改成直接返回，某些与 PatchGuard 相关的异常/清理/收尾路径会利用 KiMcaDeferredRecoveryService

Barricade

这本质上是一个先故意制造执行型缺页，再在缺页分发点筛选 PatchGuard的方法。

InjectCustomInterruptHandler() 通过自实现 trampoline hook 把 MmAccessFault 改写为 HkMmAccessFault()，

HkMmAccessFault() 只在以下条件下尝试特殊处理：

  1. 其必须由 KiPageFault调用，检查返回地址是否等于 KiPageFault一定偏移的位置，
  2. 故障类型符合 execute fault 的特征并且 TrapInformation 非空

否则就直接走原始 MmAccessFault。

如何识别PatchGuard

识别逻辑在 CustomNxFaultHandler()

从fault地址提取PML4索引，过滤掉用户模式的地址空间（索引 0~255 是用户模式地址空间）。

调用 GetPml4eVaType从内核 MiVisibleState结构中，获取该 PML4 索引对应的内核虚拟地址类型。

直接忽略 3 类地址：MiVaProcessSpace（进程空间）、MiVaDriverImages（驱动镜像空间）、MiVaPagedPool（分页池）—— 这些地址空间不会触发 PG 的 NX 异常。额外忽略页表自映射项的 PML4 索引（提取MmPteBase地址的PML4索引，判断是否相等），避免触发系统三重故障（triple fault）。

判断PTE权限位和present位

看当前 IRQL，PG 的绝大多数检查例程，都运行在 DISPATCH_LEVEL（中断请求级别 2），通过 DPC / 定时器 DPC 执行，这是 PG 的核心环境特征。

识别CmpAppendDllSection、KiDpcDispatch和KiTimerDispatch的执行路径

看 RIP 附近的机器码，匹配几类作者观察到的 PG 执行片段。

向上扫内核栈，看返回地址里是否出现 KeDelayExecutionThread、KeWaitForMultipleObjects、KeWaitForSingleObject 这类等待例程。因为验证例程完成后，一个小调度程序会在 KeDelayExecutionThread 或 KeWaitForSingleObject 之间进行选择。

PoC 使用两类处理方式：

  1. HaltNxFault 直接改 trap frame，把 RIP 弹回返回地址，相当于跳过当前故障执行点。
  2. HaltNxFault2 对线程等待场景做更复杂的 trap frame 伪造，模拟 KeDelayExecutionThread 等待。

先把哪些页改成 NX

FlipPteNxBit() / FlipPteNxBitIdPc() 多核心并行遍历页表（将256个PML4索引平均分发给每个CPU），仅对 present 页处理并对 RWX 页设置 bit 63，也就是 NX bit。因为PG 执行体代码运行在非镜像页面上，并且能实时解密，这表明会落在这些动态RWX内存上。

如何写只读代码页

WriteOnReadOnlyMemory()利用 Windows 内核的 MDL（Memory Descriptor List，内存描述符表）机制，将目标物理页面重新映射到一个新的内核虚拟地址，并显式修改新映射地址的页表权限为读写，从而绕过原虚拟地址的只读限制 。

分配 MDL，描述目标内存区域

  • MDL 是 Windows 内核中管理物理内存的核心结构，它可以记录一段虚拟地址对应的物理页面，并允许我们对这些物理页面进行重新映射、权限修改等操作。

锁定物理页面，防止被换出 MmProbeAndLockPages(mdl, KernelMode, IoReadAccess);

将物理页面重新映射到新的内核虚拟地址 mapped = MmMapLockedPagesSpecifyCache(mdl, KernelMode, MmNonCached, NULL, FALSE, NormalPagePriority);

修改新映射地址的权限为读写 MmProtectMdlSystemAddress(mdl, PAGE_READWRITE);

执行数据拷贝

这就是该 PoC 修改 MmAccessFault、KiSwInterruptDispatch、KiMcaDeferredRecoveryService 等只读内核代码页的基础能力。
